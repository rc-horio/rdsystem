# RDシステム エラーメッセージ改善方針

## 改善の3原則

1. **ユーザーにストレスを与えない** … やわらかい表現、責任を押し付けない
2. **技術的なエラーは担当者につなぐ** … 一般ユーザーに技術対応を求めない
3. **エラーの切り分けが可能なら、何のエラーか明記する** … ユーザー・担当者双方が状況を把握しやすくする

---

## エラー分類と表示方針

| 分類 | 説明 | ユーザー向け文言 | 担当者への案内 |
|------|------|-----------------|----------------|
| **入力エラー** | 入力内容・選択の不備 | 具体的に何を直すか明記。ユーザーが直せる | 不要 |
| **重複エラー** | 名前・IDの重複 | 何が重複しているか明記 | 不要 |
| **一時的な障害** | ネットワーク・API一時不通 | 時間をおいて再試行を促す | 繰り返す場合のみ |
| **データ不整合** | データ欠損・整合性の問題 | 簡潔に状況を伝える | あり |
| **システムエラー** | サーバ・設定・権限など | ユーザーは再試行のみ。詳細は担当者へ | 必須 |

## 表示フォーマットの基本形

### 入力エラー・重複エラー（ユーザー対応可能）

```
【問題の種類】〇〇を〇〇してください。
例: 入力エラー／エリア名「〇〇」は既に使われています。別の名称を入力してください。
```

### 一時的な障害（再試行で解消の可能性）

```
【一時的な障害】〇〇ができませんでした。しばらく時間をおいて、もう一度お試しください。
例: 一時的な障害／検索ができませんでした。しばらく時間をおいて、もう一度お試しください。
```

### システムエラー（担当者対応が必要）

```
〇〇ができませんでした。
担当者にお問い合わせください。
（エラー内容: プロジェクト一覧の保存に失敗）
```

技術詳細がある場合の例：

```
保存ができませんでした。
担当者にお問い合わせください。
（S3 の設定（CORS / 権限）をご確認ください）
```

- ① 何が起きたか（〇〇ができませんでした）
- ② どうするか（担当者にお問い合わせください）
- ③ 担当者向けの詳細（エラー内容や S3・CORS など）

**③ に具体的なIDを含める（推奨）**

切り分けしやすくするため、可能な限りどのプロジェクトID・エリアID・エリア名で失敗したかを書く。

```
（エラー内容: プロジェクト一覧の保存に失敗（project_id: 250101project））
（エラー内容: エリアIDが取得できません（area_name: 舞浜会場））
```

- 担当者がすぐに該当データを特定できる
- ユーザーが担当者に伝える際もそのまま伝えられる
- 取得できる情報（ID、名前など）があれば含める

**エラーコードを含める（推奨）**

担当者向けの詳細にエラーコードを付けると、切り分け・検索・ドキュメント紐づけがしやすくなる。

```
保存ができませんでした。
担当者にお問い合わせください。
（E001: プロジェクト一覧の保存に失敗（project_id: 250101project））
```

- ユーザーが「E001 が出た」と伝えるだけで担当者が状況を把握できる
- サポートチケットやログの検索・フィルタが容易
- エラーコード別にトラブルシューティングガイドを用意できる
- 命名規則の例: `E001`〜`E099` マップ、`E100`〜`E199` ハブ、`E200`〜`E299` 認証、`E300`〜`E399` ストック など

## バリデーション系（現状のまま推奨）

以下はユーザーが自分で直せる内容なので、現状のままでも問題は少ないです。

- IDを入力してください
- 7〜60文字の半角英数字のみ使用できます
- 「6桁の数字＋英字など」入力してください（例: 250101project）
- 日付が正しくありません（20YY/MM/DD は存在しません）
- このIDは既に存在します
- 保存対象のエリアを選択してください。
- エリア名を入力してください。
- 案件とスケジュールを選択してください。
- プロジェクトを選択してください

## 実装方針

### 1. メッセージの一元管理

- `packages/shared` または各アプリ内に `errorMessages.ts` を用意
- 分類・エラー種別ごとに定数化

```ts
// 例: errorMessages.ts
export const ERROR_MESSAGES = {
  // 一時的な障害
  SEARCH_TIMEOUT: "検索ができませんでした。しばらく時間をおいて、もう一度お試しください。",
  SAVE_TIMEOUT: "保存ができませんでした。しばらく時間をおいて、もう一度お試しください。",
  // システムエラー（担当者案内付き・エラーコード付き）
  SAVE_SYSTEM: (code: string, detail: string) =>
    `保存ができませんでした。\n担当者にお問い合わせください。\n（${code}: ${detail}）`,
  // ...
} as const;
```

### 2. エラー種別の明示

- エラー発生時に `reason` や `errorCode` を付与
- ユーザー向けメッセージと担当者向けの「エラー内容」を分離

```ts
// 例: エラー時に errorCode と detail を渡す
showError({
  type: "system",
  userMessage: "保存ができませんでした。",
  errorCode: "E001",
  detail: "プロジェクト一覧の保存に失敗（project_id: 250101project）",
});
```

### 3. 担当者への案内の統一

- システムエラー・データ不整合時は、共通の文言を使用
- 「担当者にお問い合わせください。」
- 問い合わせ先（メール・URL）は環境変数で管理できるようにする

### 4. 段階的な適用

1. **Phase 1**: 技術用語が多い・ストレスが大きいメッセージを優先（保存失敗、S3 など）
2. **Phase 2**: 一時的な障害の文言を統一（「失敗」→「できませんでした」など）
3. **Phase 3**: エラー分類の明示と担当者案内の追加
4. **Phase 4**: メッセージの一元管理と共通化

## 文言のトーンガイド

| 避ける | 推奨 |
|--------|------|
| 失敗しました | できませんでした |
| エラーが発生しました | 〇〇ができませんでした |
| 確認してください | お手数ですが、担当者にお問い合わせください |
| 技術詳細だけを先に出す | 担当者への案内を前段に置いてから、S3・CORS などの技術詳細を出す |
| 再試行してください | もう一度お試しください |
| 再実行してください | もう一度お試しください |
